1 Key Hierarchy

Credentials are not encrypted with the master password. A random Vault Key sits in the middle; the server stores wrapped (encrypted) copies of it.

Why the middle key? The Vault Key indirection lets a user change their master password without re-encrypting the whole vault (only the wrapped key changes) and makes recovery possible without the server ever holding a backdoor.

2 Signup Flow

Everything cryptographic happens in the browser. The server receives only random salts, a verifier, and encrypted blobs.

Browser
Action
🧑 Client
1Generate random salt, vaultKey (256-bit), recoveryKey (256-bit).
🧑 Client
2Derive keys from the master password.masterKey = Argon2id(masterPassword, salt) authHash = HKDF(masterKey, "auth")
🧑 Client
3Wrap the Vault Key twice (AES-256-GCM).wrappedVaultKey = enc(vaultKey, masterKey) wrappedRecoveryKey = enc(vaultKey, recoveryKey)
▼   POST /api/auth/register  —  salt, kdfParams, authHash, wrappedVaultKey, wrappedRecoveryKey  (no password, no plaintext key)   ▼
Server
Action
🖥️ Server
4Hash authHash again with argon2 and store the user row + blobs. Cannot derive any key from this.
▲   201 Created   ▲
Browser
Action
🧑 Client
5Display the Recovery Key once. User must confirm they saved it before the account is usable.
Critical UX The Recovery Key is shown exactly once and never reaches the server in plaintext. If the user loses it and forgets the master password, the data is unrecoverable. Say this plainly on screen.

3 Unlock Flow (Login + Decrypt)

Login proves identity with a verifier; decryption happens locally with the unwrapped Vault Key.

Browser
Action
🧑 Client
1Fetch public salt + kdfParams for the email, then derive keys.masterKey = Argon2id(masterPassword, salt) authHash = HKDF(masterKey, "auth")
▼   NextAuth authorize()  —  sends authHash (not the password)   ▼
Server
Action
🖥️ Server
2Verify argon2(authHash) against the stored hash → issue a 15-min JWT session.
▲   session + wrappedVaultKey + credential ciphertext   ▲
Browser
Action
🧑 Client
3Unwrap the Vault Key and hold it in memory for the session.vaultKey = dec(wrappedVaultKey, masterKey)
🧑 Client
4Decrypt each credential locally. Plaintext never leaves the device.{username,password,…} = dec(encryptedData, vaultKey)
Key lifetime The Vault Key lives only in browser memory. On a full page reload it is discarded and the user re-enters the master password to re-unlock (safest default).

4 Recovery Flow (Forgot Master Password)

The saved Recovery Key unwraps the same Vault Key, so a new master password can be set with zero data loss — and the server still never sees plaintext.

Browser
Action
🧑 Client
1User enters their saved Recovery Key; fetch wrappedRecoveryKey.
🧑 Client
2Unwrap the Vault Key — data is intact.vaultKey = dec(wrappedRecoveryKey, recoveryKey)
🧑 Client
3User sets a new master password → new salt, masterKey, authHash, and re-wrap.wrappedVaultKey' = enc(vaultKey, masterKey')
▼   POST new authHash + wrappedVaultKey'   ▼
Server
Action
🖥️ Server
4Replace the stored verifier + wrapped key. Never saw the Vault Key or either password.
Hard limit No Recovery Key and a forgotten master password = permanent, unrecoverable loss. This is inherent to zero-knowledge; there is no server-side backdoor by design.

5 What the Server Can & Cannot See

The credentials API becomes a dumb, secure blob store. There is no plaintext on the server for a breach to leak.

ValueOn the server?Form
Email, profile, category names✔ Yesplaintext (non-secret)
salt, kdfParams, IVs, auth tags✔ Yesplaintext (non-secret by design)
authHash (login verifier)✔ Yesargon2-hashed; cannot derive keys
wrappedVaultKey / wrappedRecoveryKey✔ Yesencrypted blob (opaque)
Credential fields (username, password, …)✔ Yesciphertext only (opaque)
Master password✗ Never—
Master Key / Recovery Key / Vault Key✗ Never—
Any credential in plaintext✗ Never—
Residual trust The one thing users still trust us with is shipping honest client code. Mitigate with Subresource Integrity, a strict CSP, dependency scanning, and a third-party audit before public launch.
